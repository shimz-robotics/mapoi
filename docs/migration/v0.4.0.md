# v0.4.0 Migration Guide

> Japanese version: [v0.4.0.ja.md](./v0.4.0.ja.md)

v0.4.0 contains **multiple breaking changes**. See the `0.4.0` section of [`CHANGELOG.rst`](../../CHANGELOG.rst) for the full list. This document summarizes the three changes users are most likely to hit.

## 1. `mapoi_nav_server` → `mapoi_nav2_bridge` rename ([#204](https://github.com/shimz-robotics/mapoi/issues/204))

The Nav2 bridge node was renamed from `mapoi_nav_server` to `mapoi_nav2_bridge`. This is a breaking change with no compatibility alias.

| Kind | Old | New |
|---|---|---|
| executable / node name | `mapoi_nav_server` | `mapoi_nav2_bridge` |
| launch arg of `mapoi_bringup.launch.yaml` | `with_nav_server` | `with_nav2_bridge` |
| C++ class | `MapoiNavServer` | `MapoiNav2Bridge` |
| header include | `mapoi_server/mapoi_nav_server.hpp` | `mapoi_server/mapoi_nav2_bridge.hpp` |
| ROS 2 services auto-derived from the node name (`/<node>/get_parameters` etc.) | `/mapoi_nav_server/...` | `/mapoi_nav2_bridge/...` |

ROS 2 launch **does not treat unknown args as fatal — it silently applies the defaults**, so if a downstream launch passes `with_nav_server:=false`, `mapoi_nav2_bridge` can start unintentionally, resulting in a silent regression. Run the audit below before updating:

```sh
grep -rn 'with_nav_server:=\|mapoi_nav_server' --include='*.launch.*'
```

ROS 2 topics / services / explicitly declared parameters (`maps_path` / `map_name` / `config_file` etc.) / Nav2 native actions are unchanged.

## 2. AMCL adapter split out into `mapoi_amcl_localization_bridge` ([#209](https://github.com/shimz-robotics/mapoi/issues/209))

The AMCL adapter was extracted from the former `mapoi_nav_server` (now `mapoi_nav2_bridge`) into a standalone `mapoi_amcl_localization_bridge` executable. `mapoi_nav2_bridge` is now a Nav2-only navigation bridge and no longer publishes to `/initialpose`.

If you launch nodes individually, add `mapoi_amcl_localization_bridge` alongside them (`mapoi_bringup.launch.yaml` already handles this via the new launch arg `with_amcl_localization_bridge`, default `true`). Parameters moved over from `mapoi_nav_server`:

- `initial_pose_topic`
- `initialpose_retry_interval_sec`
- `initialpose_retry_max_attempts`
- `initialpose_post_subscribe_republish_count`

New contract msg: `mapoi_interfaces/LocalizationBackendStatus`; new topic: `mapoi/localization/backend_status` (`transient_local` + `MANUAL_BY_TOPIC` liveliness, 5 s lease). The WebUI / RViz panel now show a **Localization indicator** independently of the Navigation indicator and gate the Initial Pose UI independently. See [docs/backend-status.md](../backend-status.md) for details.

## 3. Hardcoded system tag definitions ([#191](https://github.com/shimz-robotics/mapoi/issues/191))

The definitions of the system tags (`waypoint` / `landmark` / `pause`) are now hardcoded in `mapoi_server/include/mapoi_server/system_tags.hpp` (`mapoi::kSystemTags`); the `mapoi_server/maps/tag_definitions.yaml` file and the `mapoi_server/maps/` directory were removed.

The `get_tag_definitions` service contract is unchanged (3 system tags + user tags). Standard usage (leaving system tags untouched) is unaffected. Users who overrode the system tag names / descriptions via `tag_definitions.yaml` should edit `kSystemTags` and rebuild.

Secondary changes:
- The degenerate path that fell back to an empty system tag list when `tag_definitions.yaml` was missing / malformed is gone; the 3 system tags are now guaranteed at compile time
- The `maps/` directory is gone from the package install layout `share/mapoi_server/`. External scripts referencing `share/mapoi_server/maps/` need updating
