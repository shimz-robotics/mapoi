# v0.5.0 Migration Guide

> Japanese version: [v0.5.0.ja.md](./v0.5.0.ja.md)

v0.5.0 contains **breaking changes**. See the `0.5.0` section of [`CHANGELOG.rst`](../../CHANGELOG.rst) for the full list. This document summarizes the changes users will certainly hit.

## All `mapoi_server` services gained the `mapoi/` namespace prefix ([#341](https://github.com/shimz-robotics/mapoi/issues/341))

All 8 services exposed by `mapoi_server` were renamed to carry the `mapoi/` prefix, aligning them with the topics such as `mapoi/nav/...` (moved under `mapoi/` in v0.3.0). This is a breaking change with no compatibility alias.

| Old service name | New service name | Type |
|---|---|---|
| `get_pois_info` | `mapoi/get_pois_info` | `GetPoisInfo` |
| `get_route_pois` | `mapoi/get_route_pois` | `GetRoutePois` |
| `get_maps_info` | `mapoi/get_maps_info` | `GetMapsInfo` |
| `get_routes_info` | `mapoi/get_routes_info` | `GetRoutesInfo` |
| `select_map` | `mapoi/select_map` | `SelectMap` |
| `request_initial_pose` | `mapoi/request_initial_pose` | `RequestInitialPose` |
| `reload_map_info` | `mapoi/reload_map_info` | `std_srvs/Trigger` |
| `get_tag_definitions` | `mapoi/get_tag_definitions` | `GetTagDefinitions` |

Service types, request/response fields, topics, and launch parameters are unchanged. The clients in `mapoi_nav2_bridge` / `mapoi_amcl_localization_bridge` / `mapoi_rviz2_publisher` / `mapoi_rviz_plugins` (MapoiPanel / PoiEditor) / `mapoi_webui` / `mapoi_turtlebot3_example` all follow the new names as of this release.

### Affected users

- You call the above services directly via `ros2 service call` / a custom client / a monitoring script
- You do not run `mapoi_server` and implement only the **client side** of the above services yourself (custom bridge etc.)

If you run `mapoi_server` itself and use only the bundled packages (`mapoi_nav2_bridge` etc.), simply updating to this release is enough (the client sides are all renamed together).

### Audit procedure

Check that no old names are hardcoded in your downstream launch files / scripts / documentation:

```sh
grep -rn "ros2 service call\|create_client\|create_service" \
  --include='*.py' --include='*.cpp' --include='*.sh' \
  --include='*.yaml' --include='*.launch*' --include='*.md' \
  | grep -E "'get_pois_info'|\"get_pois_info\"|'get_route_pois'|\"get_route_pois\"|'get_maps_info'|\"get_maps_info\"|'get_routes_info'|\"get_routes_info\"|'select_map'|\"select_map\"|'request_initial_pose'|\"request_initial_pose\"|'reload_map_info'|\"reload_map_info\"|'get_tag_definitions'|\"get_tag_definitions\""
```

Rewrite any hits to the new names (with the `mapoi/` prefix) from the table above. A simple string substitution suffices:

```sh
sed -E -i \
  -e "s#([\"'])get_pois_info([\"'])#\1mapoi/get_pois_info\2#g" \
  -e "s#([\"'])get_route_pois([\"'])#\1mapoi/get_route_pois\2#g" \
  -e "s#([\"'])get_maps_info([\"'])#\1mapoi/get_maps_info\2#g" \
  -e "s#([\"'])get_routes_info([\"'])#\1mapoi/get_routes_info\2#g" \
  -e "s#([\"'])select_map([\"'])#\1mapoi/select_map\2#g" \
  -e "s#([\"'])request_initial_pose([\"'])#\1mapoi/request_initial_pose\2#g" \
  -e "s#([\"'])reload_map_info([\"'])#\1mapoi/reload_map_info\2#g" \
  -e "s#([\"'])get_tag_definitions([\"'])#\1mapoi/get_tag_definitions\2#g" \
  <path/to/your/file>
```

(This substitution only targets service-name strings wrapped in quotes. To avoid touching identifiers that happen to share a service name — such as the C++ method `request_initial_pose()` in `mapoi_nav2_bridge` — we recommend visually reviewing the diff before and after the substitution.)

The `ros2 param` auto-derived services (`/<node>/get_parameters` etc.) are tied to the node name and are out of scope for this change.

## `GetRoutePois.srv` response gained `success` / `error_message` ([#342](https://github.com/shimz-robotics/mapoi/issues/342))

`bool success` / `string error_message` were added at the head of the `mapoi/get_route_pois` response (the same pattern as `SelectMap.srv`). The existing `pois_list` / `landmark_pois` fields are unchanged, but since the `.srv` definition changes, **rebuilding the consumers is mandatory**.

- When no route matching `route_name` exists in the current map, the service previously returned empty `pois_list` / `landmark_pois` without any log; starting with this release it returns `success=false` with a descriptive `error_message`, and `mapoi_server` also emits a `WARN` log
- When the route exists but has 0 waypoints / landmarks, it returns `success=true` as before (this makes it distinguishable from a nonexistent route)

### Affected users

- You call `mapoi/get_route_pois` directly via `ros2 service call` / a custom client (the response type changes, so a rebuild is mandatory)
- You do not run `mapoi_server` and implement only the **client side** of `GetRoutePois` yourself

The clients bundled in this repository — `mapoi_nav2_bridge` / `mapoi_rviz2_publisher` / `mapoi_rviz_plugins` (`MapoiPanel`) — now check `response.success` as of this release. In particular, `mapoi_nav2_bridge` publishes `rejected:<route name>` to `mapoi/nav/status` when it receives `success=false` (the observable behavior when a nonexistent route is requested is the same as before; only the basis for the rejection changed from an empty list to `success`).

## Removed the `id` field from `PointOfInterest.msg` ([#338](https://github.com/shimz-robotics/mapoi/issues/338))

The `int32 id` field was removed from `mapoi_interfaces/msg/PointOfInterest`.

### Rationale for the removal

The only producer, `MapoiServer::yaml_to_poi_msg()` (`mapoi_server.cpp`), populated only `name` / `pose` / `tolerance` / `tags` / `description` and never assigned to `id`. As a result, `id` was always `0` for every POI obtained via `mapoi/get_pois_info` / `mapoi/get_route_pois` / `PoiEvent.poi`. The WebUI / REST API side also operates with `name` as the de facto unique key, and nothing referenced `id`. Freezing a field that is "a unique identifier on paper but always 0 in the implementation" into the v1.0.0 public API contract is a pattern that invites regret, so we decided to remove it while still in v0.x. The alternative of introducing a stable ID generation rule (option B in the issue) was not pursued.

### Affected users

- Because the `.msg` definition changes, **all packages depending on `mapoi_interfaces` must be rebuilt**
- Any external code reading `PointOfInterest.id` (custom bridge, monitoring script, custom UI etc.) needs to migrate to the `name` key. Nothing in this repository referenced `id` (the `.id` in rviz2_publisher etc. is `visualization_msgs::Marker.id`, which is unrelated)
- **rosbags that recorded topics containing the old `PointOfInterest`** (`PoiEvent` etc.) **cannot be replayed with nodes built against the new definition** (wire-format incompatibility). Likewise, a staged rollout mixing old and new `mapoi_interfaces` across nodes is not possible; update all related nodes to v0.5.0 at once (this applies not just to this section but to v0.5.0 as a whole, including #341 / #342)

### On the uniqueness of `name`

With `id` removed, `name` is the sole identifier. Uniqueness is enforced on the editing paths: both the RViz POI editor (`ValidatePois`) and the WebUI save API (#109) reject duplicate names. If you hand-edit the YAML into duplicates, name resolution (e.g. waypoint references from routes) uses the **first matching POI**, so keep names unique when hand-editing.

## Renamed the `SelectMap.srv` response field `initial_poi_name` to `resolved_initial_poi_name` ([#343](https://github.com/shimz-robotics/mapoi/issues/343))

`SelectMap.srv` had a field named `initial_poi_name` on both the request and the response, making it hard to tell at a glance "the POI name specified by the caller" (request) from "the POI name actually adopted" (response). Only the response side was renamed to `resolved_initial_poi_name`. The request-side `initial_poi_name` is unchanged.

| Field | Old name | New name |
|---|---|---|
| Response | `initial_poi_name` | `resolved_initial_poi_name` |

### Affected users

- Because the `.srv` definition changes, **all packages depending on `mapoi_interfaces` must be rebuilt**
- You call `mapoi/select_map` directly via `ros2 service call` / a custom client and read `initial_poi_name` from the response
- External clients of the `mapoi_webui` REST API `POST /api/maps/select`: the key in the JSON response body also changes from `initial_poi_name` to `resolved_initial_poi_name` (the WebUI frontend itself did not reference this key and needed no changes, but external clients must update)

`mapoi_server` (which generates the response) and `mapoi_nav2_bridge` (which reads the response), both bundled in this repository, follow the new field name as of this release.

## `mapoi_webui` REST API URLs: editor/nav hierarchy split and separator unification ([#340](https://github.com/shimz-robotics/mapoi/issues/340), [#343](https://github.com/shimz-robotics/mapoi/issues/343) item 5)

`POST /api/maps/select` (switches `mapoi_server`'s editing context; does not touch Nav2) and `POST /api/nav/switch-map` (actually switches the Nav2 map of a robot in operation) have similar names, making their behavior easy to misjudge — a safety risk that could lead to an unintended map switch on a robot in operation. Before freezing the REST API for v1.0.0, the editor-side URLs were moved under `/api/editor/`. In addition, the remaining 3 URLs whose separators were inconsistently snake_case / unseparated were unified to kebab-case (item 5 of #343).

**There are no compatibility aliases. The old URLs return `404 Not Found`.**

| Old URL | New URL | Notes |
|---|---|---|
| `POST /api/maps/select` | `POST /api/editor/select-map` | Separates the editor family (`mapoi_server` editing-context switch) from `POST /api/nav/switch-map` (real-robot map switch) at the URL hierarchy level |
| `GET /api/tag_definitions` | `GET /api/tag-definitions` | Separator unified to kebab-case |
| `POST /api/custom_tags` | `POST /api/custom-tags` | Separator unified to kebab-case |
| `POST /api/nav/initialpose` | `POST /api/nav/initial-pose` | Unseparated → kebab-case |

### Affected users

- External clients that call any of the above 4 endpoints directly via curl / a custom dashboard / a custom frontend etc.
- The WebUI itself (`mapoi_webui/web/js/`) follows the new URLs as of this release, so plain standard-UI usage is unaffected

### Difference between `/api/editor/select-map` and `/api/nav/switch-map` (recap)

- `/api/editor/select-map`: only switches `mapoi_server`'s editing context (which map's POI/Route/CustomTags the WebUI edits); it does not act on the navigation map of Nav2 / the real robot. Calling it alone can create a state where the server's editing context and the map Nav2 actually navigates on have diverged
- `/api/nav/switch-map`: actually switches the Nav2 map of a robot in operation

If you want the two to match, the caller must hit both endpoints. See the REST API table in `mapoi_webui/README.md` (the "Acts on" column) for details.

## Simplified `PoiEvent.msg` from 4 event types to 3 and limited firing to ROUTE navigation ([#220](https://github.com/shimz-robotics/mapoi/issues/220), [#227](https://github.com/shimz-robotics/mapoi/issues/227))

The event types of `mapoi_interfaces/msg/PoiEvent` were simplified from the 4 types up to v0.4.0 (`EVENT_ENTER` / `EVENT_EXIT` / `EVENT_STOPPED` / `EVENT_RESUMED`; targeting every tagged POI in any nav mode) to 3 types (`EVENT_ENTER` / `EVENT_PAUSED` / `EVENT_EXIT`; targeting only POIs registered in the route (waypoints / landmarks) while navigating a ROUTE).

| Field | Old (through v0.4.0) | New (v0.5.0) |
|---|---|---|
| Constant value 1 | `EVENT_ENTER` | `EVENT_ENTER` (unchanged) |
| Constant value 2 | `EVENT_EXIT` | `EVENT_PAUSED` (new; fires when navigation stops (`cmd_vel` dwell) at a POI tagged `pause`) |
| Constant value 3 | `EVENT_STOPPED` (removed) | `EVENT_EXIT` (constant value changed from 2 to 3) |
| Constant value 4 | `EVENT_RESUMED` (removed) | (no corresponding event; observe the resume timing via `mapoi/nav/resume` / `mapoi/nav/status`) |

The firing scope also changed:

- Old: fired for any POI carrying a target tag, regardless of nav mode (IDLE / GOAL / ROUTE)
- New: fires **only during ROUTE navigation (internal state `nav_mode_ == ROUTE`; regardless of whether the route goes through the `FollowWaypoints` path that submits the whole route to Nav2 at once, or the per-waypoint `NavigateToPose` path used with `waypoint_arrival_mode=mapoi`) and only for POIs registered in the route (waypoints / landmarks)**. It never fires during GOAL navigation or IDLE

### Affected users

- Because the `.msg` definition (constant values / field structure) changes, **all packages depending on `mapoi_interfaces` must be rebuilt**
- External subscribers of `mapoi/events` that reference the numeric `event_type` or the `EVENT_STOPPED` / `EVENT_RESUMED` constants (custom bridge, monitoring script, custom UI etc.) need to migrate to the new 3 types (`EVENT_ENTER` / `EVENT_PAUSED` / `EVENT_EXIT`). In particular, if you used `EVENT_STOPPED` as "stop detection", check whether `EVENT_PAUSED` (limited to POIs tagged `pause`) can replace it. If you were detecting stop/resume during GOAL navigation, this event can no longer serve as a substitute (refer to `mapoi/nav/status`)
- rosbags containing the old `PoiEvent` cannot be interpreted correctly by nodes built against the new definition (wire-format incompatibility). Mixed rollouts of old and new `mapoi_interfaces` are likewise not possible (the same caveat as for v0.5.0 overall, including #341 / #342 / #338 / #343)

`mapoi_nav2_bridge` (the publisher), bundled in this repository, follows the new specification as of this release. As reference implementations leveraging the new specification, `audio_guide_node` / `camera_node` sample subscribers were added to `mapoi_turtlebot3_example` (see the Samples section of the CHANGELOG, #88).
