# v0.5.0 migration ガイド

v0.5.0 は **破壊的変更** を含みます。詳細は [`CHANGELOG.rst`](../../CHANGELOG.rst) の `Unreleased` (次リリースで `0.5.0` になる予定) 節を参照してください。本ドキュメントでは利用者が確実に踏む変更を要約します。

## `mapoi_server` の全 service に `mapoi/` namespace prefix を付与 ([#341](https://github.com/shimz-robotics/mapoi/issues/341))

`mapoi_server` が公開する 8 個の service 全てを、`mapoi/nav/...` 等の topic (v0.3.0 で `mapoi/` 化済み) と揃えて `mapoi/` prefix 付きに rename しました。互換 alias なしの破壊的変更です。

| 旧 service 名 | 新 service 名 | 型 |
|---|---|---|
| `get_pois_info` | `mapoi/get_pois_info` | `GetPoisInfo` |
| `get_route_pois` | `mapoi/get_route_pois` | `GetRoutePois` |
| `get_maps_info` | `mapoi/get_maps_info` | `GetMapsInfo` |
| `get_routes_info` | `mapoi/get_routes_info` | `GetRoutesInfo` |
| `select_map` | `mapoi/select_map` | `SelectMap` |
| `request_initial_pose` | `mapoi/request_initial_pose` | `RequestInitialPose` |
| `reload_map_info` | `mapoi/reload_map_info` | `std_srvs/Trigger` |
| `get_tag_definitions` | `mapoi/get_tag_definitions` | `GetTagDefinitions` |

service の型・request/response フィールド・topic・launch parameter は不変です。`mapoi_nav2_bridge` / `mapoi_amcl_localization_bridge` / `mapoi_rviz2_publisher` / `mapoi_rviz_plugins` (MapoiPanel / PoiEditor) / `mapoi_webui` / `mapoi_turtlebot3_example` の各 client は本リリースで全て新名に追従済みです。

### 影響を受けるユーザー

- 上記 service を `ros2 service call` / 独自 client / 監視スクリプトから直接叩いている場合
- `mapoi_server` を含まず、上記 service の **client 側のみ** を自前実装している場合 (custom bridge 等)

`mapoi_server` 本体だけを起動し、付属パッケージ (`mapoi_nav2_bridge` 等) のみを使う場合は本リリースへの更新だけで動作します (client 側もまとめて新名になっているため)。

### Audit 手順

下流の launch / script / ドキュメントに旧名がハードコードされていないか確認してください:

```sh
grep -rn "ros2 service call\|create_client\|create_service" \
  --include='*.py' --include='*.cpp' --include='*.sh' \
  --include='*.yaml' --include='*.launch*' --include='*.md' \
  | grep -E "'get_pois_info'|\"get_pois_info\"|'get_route_pois'|\"get_route_pois\"|'get_maps_info'|\"get_maps_info\"|'get_routes_info'|\"get_routes_info\"|'select_map'|\"select_map\"|'request_initial_pose'|\"request_initial_pose\"|'reload_map_info'|\"reload_map_info\"|'get_tag_definitions'|\"get_tag_definitions\""
```

見つかった箇所は上表の新名 (`mapoi/` prefix 付き) へ書き換えてください。単純な文字列置換で対応できます:

```sh
sed -E -i \
  -e "s#([\"'])get_pois_info([\"'])#\1mapoi/get_pois_info\2#g" \
  -e "s#([\"'])get_route_pois([\"'])#\1mapoi/get_route_pois\2#g" \
  -e "s#([\"'])get_maps_info([\"'])#\1mapoi/get_maps_info\2#g" \
  -e "s#([\"'])get_routes_info([\"'])#\1mapoi/get_routes_info\2#g" \
  -e "s#([\"'])select_map([\"'])#\1mapoi/select_map\2#g" \
  -e "s#([\"'])request_initial_pose([\"'])#\1mapoi/request_initial_pose\2#g" \
  -e "s#([\"'])reload_map_info([\"'])#\1mapoi/reload_map_info\2#g" \
  -e "s#([\"'])get_tag_definitions([\"'])#\1mapoi/get_tag_definitions\2#g" \
  <path/to/your/file>
```

(引用符で囲まれた service 名文字列だけを対象にする置換です。C++ メソッド名の `request_initial_pose()` (`mapoi_nav2_bridge`) のように service 名と同名の識別子には触れないよう、置換前後で差分を目視確認することを推奨します。)

`ros2 param` 系の auto-derived service (`/<node>/get_parameters` 等) はノード名に紐付くため本変更の対象外です。

## `GetRoutePois.srv` の response に `success` / `error_message` を追加 ([#342](https://github.com/shimz-robotics/mapoi/issues/342))

`mapoi/get_route_pois` の response 先頭に `bool success` / `string error_message` (`SelectMap.srv` と同じパターン) を追加しました。既存の `pois_list` / `landmark_pois` フィールドは不変ですが、`.srv` 定義が変わるため **利用側の再ビルドが必須** です。

- `route_name` に一致する route が現在の地図に無い場合、従来は空の `pois_list` / `landmark_pois` を返すだけでログも出しませんでしたが、本リリースからは `success=false` + 説明入りの `error_message` を返し、`mapoi_server` 側で `WARN` ログも出します
- route は存在するが waypoints / landmarks が 0 件の場合は従来どおり `success=true` (route 不存在との区別が可能になります)

### 影響を受けるユーザー

- `mapoi/get_route_pois` を `ros2 service call` / 独自 client から直接叩いている場合 (response の型が変わるため再ビルド必須)
- `mapoi_server` を含まず、`GetRoutePois` の **client 側のみ** を自前実装している場合

本リポジトリに含まれる `mapoi_nav2_bridge` / `mapoi_rviz2_publisher` / `mapoi_rviz_plugins` (`MapoiPanel`) の各 client は本リリースで `response.success` を確認するよう追従済みです。特に `mapoi_nav2_bridge` は `success=false` を受け取ると `mapoi/nav/status` に `rejected:<route名>` を publish します (存在しない route を要求した場合の可観測な挙動自体は従来と同じで、reject の判定根拠が空リストから `success` に変わっただけです)。

## `PointOfInterest.msg` から `id` フィールドを削除 ([#338](https://github.com/shimz-robotics/mapoi/issues/338))

`mapoi_interfaces/msg/PointOfInterest` の `int32 id` フィールドを削除しました。

### 削除理由

唯一の producer である `MapoiServer::yaml_to_poi_msg()` (`mapoi_server.cpp`) は `name` / `pose` / `tolerance` / `tags` / `description` のみを埋めており、`id` への代入は一度も行われていませんでした。そのため `mapoi/get_pois_info` / `mapoi/get_route_pois` / `PoiEvent.poi` 経由で得られる全 POI の `id` は常に `0` です。WebUI / REST API 側も `name` を実質的な一意キーとして運用しており、`id` は誰にも参照されていませんでした。「ドキュメント上は一意識別子、実装上は常に 0」のフィールドを v1.0.0 の公開 API 契約として固定するのは後悔しやすいパターンのため、v0.x のうちに削除する判断としました。安定した ID 生成規則を新設する案 (issue 内の案 B) は見送りました。

### 影響を受けるユーザー

- `.msg` 定義の変更のため、`mapoi_interfaces` に依存する全パッケージは **再ビルドが必須** です
- `PointOfInterest.id` を読んでいた外部コード (custom bridge、監視スクリプト、独自 UI 等) があれば、`name` キーへの移行が必要です。本リポジトリ内には `id` を参照していた箇所はありませんでした (rviz2_publisher 等の `.id` は `visualization_msgs::Marker.id` であり別物)
- 旧定義の `PointOfInterest` を含む topic (`PoiEvent` 等) を記録した **rosbag は新定義のノードでは再生できません** (ワイヤフォーマット非互換)。同様に、新旧 `mapoi_interfaces` が混在するノード群の段階的ロールアウトも不可で、v0.5.0 へは関係ノードを一括更新してください (本節に限らず #341 / #342 を含む v0.5.0 全体に言えます)

### `name` の一意性について

`id` 削除により `name` が唯一の識別子になります。一意性は編集経路で強制されます: RViz POI editor (`ValidatePois`) と WebUI の保存 API (#109) はどちらも重複 name を reject します。YAML を手編集して重複させた場合の name 解決 (route の waypoint 参照等) は**最初に一致した POI** を使うため、手編集時は name の一意性を保ってください。

## `SelectMap.srv` の response フィールド `initial_poi_name` を `resolved_initial_poi_name` に rename ([#343](https://github.com/shimz-robotics/mapoi/issues/343))

`SelectMap.srv` は request / response の両方に `initial_poi_name` という同名フィールドを持っており、「呼び出し側が指定した POI 名」(request) と「実際に採用された POI 名」(response) の区別が一読で付きにくいという問題がありました。response 側のみを `resolved_initial_poi_name` に rename しました。request 側の `initial_poi_name` は不変です。

| フィールド | 旧名 | 新名 |
|---|---|---|
| Response | `initial_poi_name` | `resolved_initial_poi_name` |

### 影響を受けるユーザー

- `.srv` 定義の変更のため、`mapoi_interfaces` に依存する全パッケージは **再ビルドが必須** です
- `mapoi/select_map` を `ros2 service call` / 独自 client から直接叩き、response の `initial_poi_name` を読んでいる場合
- `mapoi_webui` の REST API `POST /api/maps/select` を叩く外部 client: JSON response body のキーも `initial_poi_name` から `resolved_initial_poi_name` に変わります (WebUI 本体の frontend はこのキーを参照していないため追従不要でしたが、外部 client は更新が必要です)

本リポジトリに含まれる `mapoi_server` (response の生成元) と `mapoi_nav2_bridge` (response の読み手) は本リリースで新フィールド名に追従済みです。

## `mapoi_webui` REST API URL の editor/nav 階層分離・separator 統一 ([#340](https://github.com/shimz-robotics/mapoi/issues/340), [#343](https://github.com/shimz-robotics/mapoi/issues/343) 5項目目)

`POST /api/maps/select` (`mapoi_server` の編集 context 切替、Nav2 には触れない) と `POST /api/nav/switch-map` (稼働中ロボットの Nav2 map を実際に切替える) は名称が似ているため挙動を誤推測しやすく、稼働中ロボットに対する意図しない map 切替につながりうる安全上のリスクがありました。v1.0.0 で REST API を固定する前に、editor 系の URL を `/api/editor/` 配下へ分離しました。併せて、URL の区切り文字が snake_case / 無区切りで不統一だった残り 3 件を kebab-case に統一しました (#343 の 5 項目目)。

**互換 alias はありません。旧 URL は `404 Not Found` を返します。**

| 旧 URL | 新 URL | 備考 |
|---|---|---|
| `POST /api/maps/select` | `POST /api/editor/select-map` | editor 系 (`mapoi_server` の編集 context 切替) と `POST /api/nav/switch-map` (実ロボットの map 切替) を URL 階層で峻別 |
| `GET /api/tag_definitions` | `GET /api/tag-definitions` | separator を kebab-case に統一 |
| `POST /api/custom_tags` | `POST /api/custom-tags` | separator を kebab-case に統一 |
| `POST /api/nav/initialpose` | `POST /api/nav/initial-pose` | 無区切り → kebab-case |

### 影響を受けるユーザー

- 上記 4 endpoint のいずれかを curl / 独自 dashboard / 独自 frontend 等から直接叩いている外部 client
- WebUI 本体 (`mapoi_webui/web/js/`) は本リリースで新 URL に追従済みのため、標準 UI の利用のみであれば影響はありません

### `/api/editor/select-map` と `/api/nav/switch-map` の違い (再掲)

- `/api/editor/select-map`: `mapoi_server` の編集 context (WebUI がどの地図の POI/Route/CustomTags を編集するか) を切替えるだけで、Nav2 / 実ロボットの走行地図には作用しません。単独で叩くと、server の編集 context と Nav2 の実際の走行地図が乖離した状態を作れます
- `/api/nav/switch-map`: 稼働中ロボットの Nav2 map を実際に切替えます

両者を一致させたい場合は呼び出し側が両方の endpoint を叩く必要があります。詳細は `mapoi_webui/README.md` の REST API 表 (「作用対象」列) を参照してください。
